# PRD: 경제 뉴스 자동 브리핑 기능

- 문서 버전: v1.0
- 작성일: 2026-07-24
- 대상 서비스: 간편 알람 (Simple Alarm) 웹앱
- 근거 문서: `기술명세서.md` (경제 뉴스 자동 프리핑 기능 추가 명세, v1.0)

---

## 1. 배경 및 목표

### 1.1 배경

간편 알람 웹앱은 현재 알람 등록/관리, 실시간 시각 표시, 서울 현재 기온 표시 기능을 제공하고 있다.
사용자가 알람으로 기상하는 시점에 시사·경제 정보를 함께 확인할 수 있도록, 최근 경제 뉴스 5개를 자동으로 불러와 보여주는 기능을 추가한다.

날씨 API와 달리 대부분의 뉴스 소스(언론사 RSS, 네이버 뉴스 검색 API 등)는 브라우저에서 직접 호출 시 CORS 정책에 의해 차단되므로, 별도의 프록시 또는 서버리스 백엔드 경유 구조가 필요하다. 특히 네이버 뉴스 검색 API는 Client ID/Secret 인증이 필수이므로 서버리스 백엔드 경유가 불가피하다.

### 1.2 목표

- 페이지 진입 시 최근 경제 뉴스 5개를 자동으로 불러와 표시한다(프리핑).
- 일정 주기(기본 10분)마다 뉴스를 자동 갱신한다.
- 뉴스 제목 클릭 시 새 탭에서 원문 기사로 이동한다.
- 초기에는 프록시 기반(방법 A)으로 빠르게 구현하고, 실서비스/배포 단계 또는 키워드 검색이 필요한 시점에는 서버리스 백엔드 기반의 RSS(방법 B) 또는 네이버 뉴스 검색 API(방법 C)로 전환 가능한 구조를 지향한다.

---

## 2. 사용자 스토리

- 사용자로서, 알람이 울려 화면을 켰을 때 최근 경제 뉴스 5개를 바로 확인하고 싶다. 아침에 다른 뉴스 앱을 따로 열지 않아도 되기 때문이다.
- 사용자로서, 관심 있는 뉴스 제목을 클릭하면 새 탭에서 원문 기사를 바로 읽고 싶다. 앱을 벗어나지 않고 원문을 확인하고 싶기 때문이다.
- 사용자로서, 뉴스가 자동으로 최신 상태로 갱신되길 원한다. 직접 새로고침하지 않아도 최근 소식을 놓치지 않고 싶기 때문이다.
- 사용자로서, 뉴스 로딩이 실패하더라도 알람 소리는 반드시 정상적으로 울리길 원한다. 뉴스는 부가 기능이고 알람의 본질적 기능(기상)은 어떤 경우에도 보장되어야 하기 때문이다.

## 3. 기능 요구사항

### 3.1 뉴스 표시

- FR-1. 페이지 로딩 시 최근 경제 뉴스 최대 5개를 자동으로 불러와 화면에 표시한다.
- FR-2. 각 뉴스 항목은 제목(title)과 원문 링크(link)로 구성된다.
- FR-3. 뉴스 제목 클릭 시 `target="_blank"`, `rel="noopener"` 속성으로 새 탭에서 원문 기사로 이동한다.
- FR-4. 뉴스 목록은 온도 표시 박스 하단에 카드 형태(`#news-box`)로 노출한다.

### 3.2 자동 갱신

- FR-5. 최초 진입 시 즉시 1회 뉴스를 조회한다.
- FR-6. 이후 기본 10분(600,000ms) 주기로 자동 재조회하여 목록을 갱신한다.

### 3.3 데이터 소스 연동

- FR-7. 뉴스 데이터를 다음 세 방식 중 하나로 조회한다.
  - 방법 A: 브라우저 → 무료 프록시(예: rss2json) → 언론사 RSS
  - 방법 B: 브라우저 → 서버리스 함수(Vercel/Netlify/Cloudflare Workers) → 언론사 RSS
  - 방법 C: 브라우저 → 서버리스 함수 → 네이버 뉴스 검색 API(`https://openapi.naver.com/v1/search/news.json`), 서버 측에서 `X-Naver-Client-Id`/`X-Naver-Client-Secret` 헤더로 인증
- FR-8. 초기 구현은 방법 A를 채택하고, 실서비스 전환 시 방법 B 또는 방법 C로 교체 가능해야 한다(데이터 소스 교체가 프론트 표시 로직에 영향을 주지 않도록 인터페이스를 분리한다).
- FR-11. 방법 C 채택 시 네이버 뉴스 검색 API 응답의 `title`, `description` 필드에 포함된 HTML 태그(`<b>` 등 하이라이트 태그 포함)를 제거한 뒤 표시한다.

### 3.4 예외 처리

- FR-9. 뉴스 목록이 비어 있는 경우 "뉴스를 가져오지 못했습니다." 메시지를 표시한다.
- FR-10. 네트워크 오류 등으로 조회 자체가 실패한 경우 "뉴스를 불러오지 못했습니다." 메시지를 표시한다.

## 4. 비기능 요구사항 (보안 최우선)

### 4.1 보안

- **NFR-SEC-1. 네이버 뉴스 검색 API(방법 C) 사용 시 Client ID/Secret(`NAVER_CLIENT_ID`, `NAVER_CLIENT_SECRET`)은 어떠한 경우에도 프론트엔드(HTML/JS 소스, 브라우저 네트워크 탭, 클라이언트 번들)에 노출되어서는 안 된다.** 반드시 서버리스 함수의 환경 변수로만 저장·관리하며, 클라이언트에서 네이버 API를 직접 호출하는 구조는 허용하지 않는다. 방법 B(RSS 서버리스)에 향후 별도 인증이 필요한 뉴스 API를 도입하는 경우에도 동일 원칙을 적용한다. 방법 A(무료 RSS 프록시)는 별도 인증 키가 불필요하다.
- NFR-SEC-2. 서버리스 함수(방법 B/C) 사용 시 CORS 헤더(`Access-Control-Allow-Origin`)는 필요한 범위로 최소화하여 설정한다.
- NFR-SEC-3. 언론사 기사 본문을 그대로 재게시하지 않는다. 제목과 원문 링크만 노출하여 저작권 침해 소지를 최소화한다.
- NFR-SEC-4. 사용자 입력값(있는 경우) 및 API 응답에 포함된 텍스트(뉴스 제목 등)는 XSS 방지를 위해 이스케이프 처리 후 렌더링한다.
- NFR-SEC-5. 네이버 개발자센터 애플리케이션의 API 키는 정기적으로 로테이션 가능한 구조로 관리하고, 저장소(git 등)에 커밋되지 않도록 `.gitignore`/환경 변수로 분리한다.

### 4.2 데이터 관리

- **NFR-DATA-1. 뉴스 데이터는 DB 또는 서버 측 영속 저장소에 저장하지 않는다. 매 요청마다 실시간으로 원본 소스(RSS/프록시/API)에서 조회한다.** (서버리스 함수의 짧은 캐시 헤더(`Cache-Control: s-maxage`)는 CDN/엣지 단의 일시적 캐시로만 허용하며, 별도 DB 적재는 하지 않는다.)

### 4.3 안정성/가용성

- **NFR-AVAIL-1. 뉴스 브리핑 기능(조회, 갱신, 렌더링)이 어떤 이유로든 실패(네트워크 오류, API 오류, 파싱 오류 등)하더라도 알람 소리 자체는 반드시 정상적으로 울려야 한다.** 뉴스 로딩 로직은 알람 재생 로직과 독립적으로 동작해야 하며, 뉴스 관련 예외가 알람 타이머/재생 흐름에 영향을 주어서는 안 된다.
- NFR-AVAIL-2. 뉴스 조회 실패 시 앱의 다른 기능(알람 등록/관리, 시각/기온 표시)은 정상 동작을 유지한다.
- NFR-AVAIL-3. 무료 프록시(방법 A)의 호출 횟수/속도 제한을 고려하여, 과도한 요청(예: 짧은 주기 반복 새로고침)으로 인한 차단 가능성을 최소화한다.
- NFR-AVAIL-4. 네이버 뉴스 검색 API(방법 C)의 일 호출 한도(애플리케이션당 일 25,000회, 정책 변경 가능)를 초과하지 않도록 자동 갱신 주기 및 캐싱을 고려한다.

### 4.4 성능

- NFR-PERF-1. 뉴스 조회는 페이지의 다른 기능(시계, 알람 목록) 렌더링을 블로킹하지 않는다(비동기 처리).

## 6. Open Questions

아래 항목은 기술명세서.md에 미확정으로 명시되어 있으며, 확정된 것처럼 구현해서는 안 된다. 별도 논의 및 의사결정이 필요하다.

- **OQ-1. 알람-뉴스 브리핑 연동 재생 타이밍 미확정 (기술명세서.md 9.6절).** 알람 소리와 뉴스 브리핑 표시가 동시에 시작되는지, 알람 정지 이후에 표시되는지 순서가 미정이다. 스누즈 시 브리핑을 매번 갱신할지 여부, 향후 음성(TTS) 브리핑 도입 시 알람 소리와의 재생 순서/딜레이도 미정이다.
- **OQ-2. 뉴스 검색 키워드 미확정 (기술명세서.md 10절).** 방법 C(네이버 뉴스 검색 API) 채택 시 사용할 키워드가 고정 단일 키워드("경제")인지 다중 키워드 조합인지, 정렬 기준을 관련도순(`sim`)/최신순(`date`) 중 무엇으로 할지, 사용자가 키워드를 직접 설정할 수 있게 할지가 결정되지 않았다.
- OQ-3. 방법 A(프록시) / 방법 B(RSS 서버리스) / 방법 C(네이버 API 서버리스) 중 최종 채택 방식 및 전환 시점·트리거 기준 미확정 (트래픽 임계치, 배포 시점, 키워드 검색 필요 여부 등).
- OQ-4. 최종적으로 사용할 언론사/RSS 소스 미확정 (한국경제 vs 매일경제 vs 복수 소스, 각 언론사 RSS 이용 약관 검토 필요).
- OQ-5. 자동 갱신 주기(기본 10분)를 사용자가 변경 가능하게 할지 여부.
- OQ-6. 네이버 개발자센터 애플리케이션 등록 및 Client ID/Secret 발급 주체·관리 방식(누가 발급받고 어떤 환경 변수 저장소에 보관할지) 미정.

## 7. 성공 기준

- 페이지 진입 시 3초 이내에 최근 경제 뉴스 5개(또는 실패 메시지)가 표시된다.
- 뉴스 제목 클릭 시 새 탭에서 원문 기사로 정상 이동한다.
- 10분 주기 자동 갱신이 별도 조작 없이 동작한다.
- 뉴스 조회 실패를 인위적으로 유발(네트워크 차단 등)했을 때도 알람 소리는 설정된 시각에 정상적으로 울린다.
- 브라우저 네트워크 탭/소스 코드/클라이언트 번들 검사에서 네이버 Client ID/Secret 등 API 키·인증 토큰이 노출되지 않는다(방법 B/C 적용 시).
- 서버(또는 서버리스 함수) 측에 뉴스 데이터가 DB로 영속 저장되지 않고, 매 요청마다 원본 소스 조회가 발생함을 확인한다.
