기능별 세부 규칙(파일 구조, 함수 이름 등)은 여기 넣지 않고 각 기능 폴더의 CLAUDE.md에 별도로 둔다.

## 🚨 최우선 보안 규칙: API 키/시크릿을 프론트엔드에 절대 넣지 않는다

**네이버 API Client ID/Secret을 포함한 모든 외부 API 키·인증 토큰·시크릿은 어떤 경우에도 프론트엔드 코드(HTML, JS, 브라우저에서 실행되는 코드, 클라이언트 번들)에 작성하거나 노출해서는 안 된다.**

- 키가 필요한 외부 API 호출은 반드시 서버(서버리스 함수 등)를 경유하며, 키는 서버 측 환경 변수로만 저장·관리한다.
- 브라우저 개발자도구의 네트워크 탭, 소스 코드 보기, 클라이언트 번들 어디에서도 키/시크릿 문자열이 보이면 안 된다.
- 키를 `.env` 등 파일에 두는 경우 반드시 `.gitignore`에 포함하여 저장소(git)에 커밋되지 않도록 한다.
- 새로운 외부 API를 연동할 때도 이 원칙을 예외 없이 동일하게 적용한다.

---

## 보안 및 아키텍처 원칙

- **프론트/백엔드 분리:** 인증이 필요하거나 CORS로 막히는 외부 API는 브라우저에서 직접 호출하지 않고, 서버리스 함수 등 서버 계층을 경유시킨다.
- **CORS 최소화:** 서버 측에서 CORS 헤더(`Access-Control-Allow-Origin` 등)를 설정할 때는 필요한 범위로만 최소화한다.
- **데이터 비저장(무상태 조회) 원칙:** 외부에서 가져오는 데이터(뉴스 등)는 DB나 서버 측 영속 저장소에 저장하지 않고, 매 요청마다 원본 소스에서 실시간으로 조회한다. CDN/엣지 단의 짧은 캐시(`Cache-Control` 등)는 허용하되, 별도 DB 적재는 하지 않는다.
- **입력/출력 이스케이프:** 사용자 입력값과 외부 API 응답에 포함된 텍스트는 렌더링 전 XSS 방지를 위해 이스케이프 처리한다.
- **저작권 최소 노출:** 외부 콘텐츠(기사 등)는 본문을 그대로 재게시하지 않고, 제목/링크 등 최소한의 정보만 노출한다.

## 실패 허용(Graceful Degradation) 원칙

**부가 기능이 실패해도 핵심 기능은 반드시 정상 동작해야 한다.** 대표적으로, 뉴스 브리핑 등 부가 기능이 어떤 이유로든 실패(네트워크 오류, API 오류, 파싱 오류 등)하더라도 **알람 소리 자체는 반드시 정상적으로 울려야 한다.**

- 부가 기능의 로직은 핵심 기능(알람 재생 등)의 로직과 독립적으로 동작하도록 분리한다.
- 부가 기능에서 발생한 예외가 핵심 기능의 타이머/실행 흐름에 전파되어 영향을 주어서는 안 된다.
- 부가 기능 실패 시에는 해당 영역에만 실패 안내를 표시하고, 나머지 기능(알람 등록/관리, 시각/기온 표시 등)은 정상 동작을 유지한다.

## 작업 순서 원칙

새로운 기능을 구현할 때는 다음 순서를 지킨다.

1. **화면(UI)을 가짜 데이터(mock data)로 먼저 만든다.** 실제 API 연동 없이 레이아웃과 표시 로직부터 완성한다.
2. **그다음 실제 백엔드/외부 API를 연결한다.** 가짜 데이터를 실제 데이터 조회 로직으로 교체한다.
3. **예외 처리는 가장 마지막에 추가한다.** 정상 동작 경로가 확인된 뒤에 실패/에러 케이스에 대한 처리를 붙인다.

이 순서를 임의로 건너뛰거나 뒤바꾸지 않는다.

## 미확정 사항 처리 원칙

PRD의 Open Questions로 분류된 항목(예: 뉴스 검색 키워드 방식, 알람-브리핑 재생 타이밍)은 **아직 확정되지 않았다.** 이런 항목을 구현할 때 임의로 값을 정해서 확정된 것처럼 코드에 반영하지 않는다. 관련 작업이 필요하면 먼저 사용자에게 확인하거나, 명확히 "미확정/임시" 상태임을 표시한다.
